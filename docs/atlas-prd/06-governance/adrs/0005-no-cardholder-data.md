# ADR 0005 — Atlas does not store, process, or transmit real cardholder data

- **Status:** Accepted
- **Date:** 2026-07-20
- **Related requirements:** FND-012, KYC-003, POR-003

## Context

Cardholder data would materially increase security, compliance, legal, infrastructure, and operational obligations while contributing little additional signal to the selected wallet/ledger/reconciliation portfolio thesis. A superficial card form would create misleading PCI claims.

## Decision

- No real PAN, CVV, track, PIN, cardholder, or payment-card authentication data.
- No production card network or processor integration.
- If a payment-method UI is needed, it uses synthetic tokens generated by the provider simulator.
- Logs, fixtures, documentation, and screenshots use unmistakably synthetic values.
- Security scans include patterns/canaries to detect accidental card-like data.
- Documentation states that PCI DSS is a scope-awareness reference only and Atlas does not claim compliance.

## Consequences

Effort remains focused on ledger, payment lifecycle, provider ambiguity, merchant APIs, webhooks, settlement, reconciliation, risk, and operations. A future card-issuing simulator would be a separate scoped project/ADR.

## Rejected alternatives

- Storing test PANs “because they are not real.”
- Embedding a third-party checkout and implying card-system expertise.
- Claiming PCI compliance from avoiding a few fields.

## Verification

Repository/image/object-store scan; public demo data review; simulator token contract; release claims review.
