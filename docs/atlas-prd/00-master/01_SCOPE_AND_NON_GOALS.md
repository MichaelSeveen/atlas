# Scope and non-goals

## 1. Scope boundary

Atlas models the core control plane of a wallet and payments company. It is broad enough to demonstrate end-to-end financial operations, but intentionally excludes entire industries that would dilute the signal or create false claims.

## 2. In-scope currencies and money model

- Initial currencies: NGN and USD.
- Every account is single-currency.
- Currency exponent is stored as reference data and is never inferred from display formatting.
- Amounts are integer minor units in persistence and APIs.
- FX is simulated through expiring quotes, explicit spread, and linked journals. No live trading or treasury execution is required.

## 3. In-scope rails

- Internal book transfer between Atlas wallets.
- Simulated bank credit into an Atlas clearing account.
- Simulated bank payout through provider adapters.
- Merchant payment intent backed by internal or simulated provider payment.
- Settlement files generated by the provider simulator.

## 4. In-scope product surfaces

- Customer web application.
- Merchant dashboard.
- Merchant developer portal.
- Risk and compliance workbench.
- Operations and support console.
- Finance and reconciliation console.
- Security and audit evidence view.

## 5. Explicit non-goals

### No real money movement

All providers are deterministic simulators. A production-looking UI must display an environment banner and synthetic-data designation.

### No cardholder-data environment

Do not collect or persist PAN, CVV, track data, PIN, or real card credentials. A card-network simulator is a future separate project, not part of Atlas.

### No native mobile applications

A responsive web application is sufficient. Building iOS and Android clients would multiply UI work without increasing the core financial-system signal.

### No React and Vue duplication

Choose one frontend framework. The reference is React. Reimplementing the same product in Vue is not a meaningful enterprise capability.

### No microservices theatre

Do not create one service per noun. Start with bounded modules in a modular monolith, one database cluster, and separate API/worker processes. Extraction requires an ADR backed by evidence.

### No custom identity provider

Do not build password storage, MFA enrollment, passkey verification, token issuance, or federation protocols from scratch. Integrate a standards-compliant identity provider and demonstrate secure application integration, session management, step-up, and authorization.

### No custom cryptographic primitives

Use mature libraries and a key-management abstraction. Do not invent encryption formats, token algorithms, signature schemes, password hashing, or random-number generators.

### No speculative AI decision maker

No LLM may approve, decline, freeze, reverse, or move money. A later bounded summarisation assistant may create a draft case summary, but it cannot be a core requirement and must be isolated from secrets and decision authority.

### No blockchain or stablecoin path

Cross-border and digital-asset settlement are separate portfolio projects. They would obscure the primary ledger, reconciliation, and operational objectives here.

### No vanity dashboards

Every chart must answer an operational, financial, risk, customer, or reliability question. Decorative KPI cards are not deliverables.

### No fake “exactly once” claim

Transport is assumed at-least-once. Exactly-once business effects are achieved through idempotency, unique constraints, inbox/outbox records, and transactional state transitions.

### No unsupported compliance claims

The project is standards-aligned and control-mapped. It is not certified or audited. Use “designed against” or “mapped to,” never “compliant with,” unless an authorized assessment exists.

### No premature multi-region active-active

Design region failure assumptions, backups, and recovery. Do not implement cross-region active-active ledger writes without a justified consistency model and an operational team to support it.

### No event sourcing for every domain

The financial journal and audit history are append-only. Ordinary profile and configuration domains may use conventional state tables plus audit events. Do not force event sourcing where it adds complexity without value.

## 6. Deferred extensions

The following can be documented as future work but must not delay the core project:

- card issuing and ISO 8583 simulation;
- lending, credit scoring, or underwriting;
- real-time gross settlement integrations;
- agent banking;
- stablecoin settlement;
- open-banking consent journeys;
- machine-learning fraud models;
- native mobile applications;
- multi-region write availability;
- general-purpose workflow engine.

## 7. Scope change policy

A proposed feature enters scope only when it:

1. strengthens one of the core pillars: correctness, security, risk, operations, reconciliation, or explainability;
2. has defined users and measurable acceptance criteria;
3. has a threat and failure analysis;
4. does not weaken the phase sequence;
5. does not replace a core capability with a third-party demo wrapper;
6. has an explicit “what will not be built” section.
