# Merchant webhook security and delivery specification

## Goal

Deliver merchant-facing events with strong authenticity, replay resistance, bounded retries, safe destination validation, operational transparency, and no duplicate financial effect requirement on the merchant.

## Endpoint registration

A merchant may register an HTTPS endpoint only after:

1. authenticated tenant-scoped request;
2. step-up authentication;
3. authorization for `webhook_endpoints:create`;
4. URL canonicalization and network policy validation;
5. ownership challenge;
6. successful test event;
7. immutable audit record.

### Destination policy

Reject:

- non-HTTPS schemes;
- userinfo in URL;
- fragments;
- literal or DNS-resolved loopback, link-local, multicast, carrier-grade NAT, private, metadata, and organization-internal ranges;
- non-standard ports unless explicitly allowlisted;
- DNS names that change from public to prohibited ranges between validation and connection;
- redirects to a different origin or prohibited address;
- destinations whose certificates fail normal validation.

Resolve and connect through a constrained egress component. Pin the validated IP only for the connection attempt while preserving correct TLS hostname verification. Re-evaluate DNS for each attempt and after redirects; redirects are disabled by default.

## Signing model

Atlas signs the exact transmitted bytes. The default profile uses HTTP Message Signatures concepts with an Atlas-defined interoperability profile:

Signed components:

- `@method`
- `@target-uri`
- `content-digest`
- `content-type`
- `atlas-event-id`
- `atlas-delivery-id`
- `atlas-timestamp`

Headers:

```http
Content-Type: application/json
Content-Digest: sha-256=:BASE64_DIGEST:
Atlas-Event-Id: evt_...
Atlas-Delivery-Id: whd_...
Atlas-Timestamp: 2026-07-20T15:00:00Z
Signature-Input: atlas=("@method" "@target-uri" "content-digest" "content-type" "atlas-event-id" "atlas-delivery-id" "atlas-timestamp");created=...;keyid="whk_...";alg="hmac-sha256"
Signature: atlas=:BASE64_SIGNATURE:
```

Portfolio reference implementation may use HMAC-SHA-256 with independent per-endpoint secret versions. The design isolates the algorithm so an asymmetric profile can be added without changing event semantics.

### Secret handling

- Secret is generated by a CSPRNG, shown once, encrypted with a managed-key abstraction, and never logged.
- Rotation creates a new key version and configurable overlap window.
- Deliveries identify signing key ID/version but never expose key material.
- Revocation can be immediate; in-flight retries re-sign with policy-selected active key and record the change.
- Workforce cannot reveal existing secret; it can only rotate/revoke under permission and audit.

## Replay resistance

Merchant verification must:

1. parse timestamps strictly;
2. enforce a documented clock-skew window;
3. compute content digest from raw bytes;
4. reconstruct the covered components exactly;
5. select the referenced active/recent key version;
6. compare signatures in constant time;
7. store processed event or delivery IDs under a durable retention period;
8. accept legitimate retries as duplicates but execute business effect once.

Atlas never assumes a `2xx` proves the merchant processed the event exactly once.

## Event body

```json
{
  "id": "evt_01J...",
  "type": "payment_intent.succeeded.v1",
  "created_at": "2026-07-20T15:00:00.123Z",
  "livemode": false,
  "merchant_id": "mer_01J...",
  "api_version": "2026-07-20",
  "data": {
    "object": {
      "id": "pi_01J...",
      "amount": {"amount_minor": "250000", "currency": "NGN"},
      "status": "succeeded"
    }
  }
}
```

The snapshot is allowlisted. Merchants query the API for current state; the webhook proves a point-in-time event.

## Delivery state machine

`scheduled -> attempting -> succeeded`

Failure branches:

- `attempting -> retry_scheduled`
- `attempting -> exhausted`
- `attempting -> endpoint_disabled`
- manual replay creates a new delivery linked to original, not mutation of history.

## Retry policy

- Bounded exponential backoff with jitter.
- Retry network failure, timeout, `408`, `425`, `429`, and selected `5xx`.
- Do not retry ordinary `2xx`, `3xx` (redirects disabled), most `4xx`, invalid certificate, or destination-policy failure without operator/merchant change.
- Honor safe bounded `Retry-After` but cap it.
- Concurrency and rate limits are per endpoint and merchant to avoid a noisy tenant exhausting delivery workers.
- Every attempt has deadline, maximum response bytes, response-body redaction/truncation, and connection limits.

## Manual replay

Requirements:

- tenant permission and step-up;
- event remains eligible and not beyond retention;
- explicit destination and API version;
- reason and audit;
- maximum replay window/count;
- creates new delivery ID, preserves event ID;
- cannot alter payload to make an old event say something new;
- supports dry-run signature preview only with synthetic data.

## Operational UI

Show:

- endpoint status, URL host with safe masking, event subscriptions, API/signature version;
- key versions and rotation dates without secret;
- delivery timeline and attempt count;
- request headers/body under permission and redaction;
- response status, safe headers, truncated/redacted body, latency;
- next retry and normalized failure;
- replay action with approval policy;
- endpoint health and disable reason.

## Tests most teams miss

1. Domain resolves publicly at registration then to `169.254.169.254` at delivery.
2. Redirect chain lands on private network.
3. IPv6 loopback, IPv4-mapped IPv6, octal/decimal IP forms, trailing-dot host, mixed-case/punycode confusion.
4. DNS response changes between validation and connect.
5. TLS SNI/hostname verification is preserved while connecting to validated IP.
6. Body is compressed/decompressed differently; signature still covers transmitted bytes.
7. JSON field reordering/whitespace changes invalidate signature because raw body is signed.
8. Duplicate event with a new delivery ID is deduplicated by merchant fixture.
9. Same delivery ID with altered body fails digest/signature.
10. Timestamp exactly inside/outside skew boundaries.
11. Key rotates after scheduling but before attempt; selected policy is deterministic and auditable.
12. Old key revoked during retry series.
13. Worker crashes after merchant receives request but before success is persisted; retry occurs and merchant effect remains once.
14. Merchant returns an unbounded response body or slow-drips headers/body.
15. Endpoint returns `429` with absurd or malformed `Retry-After`.
16. One endpoint stalls all connections; tenant/global bulkheads remain healthy.
17. Manual replay races automatic retry.
18. Attempt logs prove redaction against secrets and malicious newline/header values.
19. Event subscription changes while old events remain queued.
20. API version upgrade preserves old event fixtures and signatures.

## Metrics and alerts

- delivery success by endpoint/event type;
- attempt latency and timeout class;
- retry queue age;
- exhausted deliveries;
- signature key version usage;
- destination-policy blocks;
- endpoints disabled automatically;
- worker pool saturation;
- per-tenant backlog;
- manual replay volume and failures.

Alert on broad merchant failure spikes, queue age beyond SLO, key-decryption failures, unexpected destination policy blocks, and any evidence of internal-address connection.
